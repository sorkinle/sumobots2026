# FR-016: Tunable-Parameter Console Commands

## Requirement

**Statement:** The firmware shall accept the following text commands on the USB-serial console (each terminated by newline):

| Command | Effect |
|---------|--------|
| `get <name>` | Print the current value of parameter `<name>` |
| `set <name> <value>` | Set parameter `<name>` to `<value>` in RAM (not persisted) |
| `dump` | Print all parameter names and current values |
| `save` | Persist all current parameter values to NVS |
| `load` | Re-read parameter values from NVS, discarding RAM-only edits |
| `forget` | Clear the paired-controller BLE address from NVS |
| `status` | Emit a one-shot state snapshot (see FR-015) |

The minimum tunable parameter set is:
`mapping_mode`, `deadband`, `max_power`, `slew`, `curve_type`, `curve_factor`, `vbatt_threshold`, `vbatt_filter_ms`, `linkloss_timeout_ms`, `arm_combo`, `arm_hold_ms`, `telemetry_rate`.

**Priority:** Should Have
**Status:** Draft

## Traceability

| Traces To | ID | Description |
|-----------|-----|-------------|
| Customer Need | CN.12 | Expose tunable parameters via a serial command interface |
| Customer Problem | CP.8 | Tune driving parameters without recompiling and re-flashing |

## Acceptance Criteria

- [ ] All seven commands implemented and produce documented output
- [ ] `set` on a parameter that affects mixing takes effect within one input loop cycle
- [ ] Invalid parameter names produce a descriptive error and no state change
- [ ] Out-of-range values (e.g., `set deadband 1.5`) produce a descriptive error and no state change
- [ ] All listed parameters are settable via `set`

## Notes

`save` is a separate explicit step so a pilot can experiment freely in RAM and roll back via `load` if a setting feels wrong.
