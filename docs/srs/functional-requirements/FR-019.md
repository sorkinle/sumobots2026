# FR-019: Fault-Clear Action

## Requirement

**Statement:** While in **FAULT** mode, the firmware shall remain in FAULT until the operator holds the Xbox **Menu** button for ≥ **2.0 s**. Upon a successful 2.0 s hold, the firmware shall re-evaluate the underlying fault condition; if it no longer holds (e.g., battery voltage has recovered above `vbatt_threshold` plus a small hysteresis margin), the firmware shall transition to **DISARMED**. If the fault condition still holds, the firmware shall remain in FAULT and log the persistent fault to the serial console.

**Priority:** Should Have
**Status:** Draft

| Customer Problem | CP.5 | The robot must protect hardware from foreseeable failure modes |

## Acceptance Criteria

- [ ] In FAULT with `Vbatt < vbatt_threshold`, holding Menu 2.0 s → remains in FAULT; serial logs persistent fault
- [ ] In FAULT with Vbatt recovered above threshold + hysteresis, holding Menu 2.0 s → transitions to DISARMED with motors=0
- [ ] Releasing Menu before 2.0 s → no state change
- [ ] Fault-clear never transitions directly to ARMED (operator must re-arm via FR-005)
- [ ] LED transitions to DISARMED pattern within 1 s of successful clear

## Notes

The 2.0 s hold is intentionally longer than FR-005's 1.0 s arming hold, so the operator can't accidentally clear a fault. Hysteresis on the brownout threshold (e.g., +0.2 V over the trip threshold) prevents flutter between FAULT and DISARMED near the boundary.
