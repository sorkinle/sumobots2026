# NFR-004: Runtime Tunability Without Reflash

## Requirement

**Category:** Maintainability
**Statement:** Every parameter that materially influences driving "feel" or safety thresholds — `mapping_mode`, `deadband`, `max_power`, `slew`, `curve_type`, `curve_factor`, `vbatt_threshold`, `vbatt_filter_ms`, `linkloss_timeout_ms`, `arm_combo`, `arm_hold_ms`, `telemetry_rate` — shall be modifiable at runtime via the USB-serial console, take effect within one input loop cycle, and be optionally persistable via the `save` command. No "feel-affecting" parameter shall require recompiling or re-flashing the firmware to change.

**Priority:** Must Have
**Status:** Draft

## Traceability

| Traces To | ID | Description |
|-----------|-----|-------------|
| Customer Need | CN.12 | Expose tunable parameters via a serial command interface |
| Customer Need | CN.13 | Persist tuned parameters across reboots |
| Applies To FRs | FR-016, FR-017 | Console commands, NVS persistence |
| Customer Problem | CP.8 | Tune driving parameters without recompiling and re-flashing |

## Acceptance Criteria

- [ ] All listed parameters are settable via `set`
- [ ] `set` on any feel-affecting parameter takes effect within one input loop cycle (≤ 10 ms)
- [ ] All listed parameters survive `save` + power-cycle (verified by `dump`)
- [ ] No "feel-affecting" parameter requires a rebuild to change

## Measurement Method

Walk through each parameter at the serial console; verify the behavior change and the persistence.
